## 1. Workflow 0 – Audience → Drafts → Send

### W0.v0 – Baseline Segment & Draft Flow (MVP)

**Goal:** Minimal path from DB → segment → drafts → send.

**Scope:**
- Manual filter builder on companies/employees.
- One immutable segment (no versioning) per filter definition.
- One or two base email templates per segment (intro + bump).
- Simple export / send integration.

**Key capabilities:**
1. Company & contact filters (manual UI only).
2. Save segment (no version field yet, implied `v0`).
3. Generate base intro/bump drafts for the whole segment.
4. User can manually edit drafts.
5. Dispatch via Smartlead / SMTP as a bulk action.

---

### W0.v1 – Segment Versioning & Contact-Level Drafts

**Goal:** Make audience changes and finalization explicit; attach drafts and sends to a stable snapshot.

**Additions vs v0:**

1. **Segment versioning**
   - `segment.version` field introduced.
   - First saved audience = `version = 0` (raw filters).
   - After user confirms audience for outreach → new snapshot `version = 1` (final audience).
   - Any further change to filters or inclusion rules → new row or versioned snapshot.

2. **Contact-level membership (`segment_members`)**
   - `segment_members` is materialized only for **final** versions (v1+).
   - Each row links `segment_id`, `segment_version`, `employee_id`, `inclusion_reason`.

3. **Drafts per contact**
   - For each `segment_member` we can attach:
     - `draft_email_1` (intro), `draft_email_2` (bump).
     - Metadata: `pattern_id`, `generator` (coach agent), `user_edited` flag.

4. **Send job entity**
   - Explicit `send_job` entity referencing:
     - `segment_id`, `segment_version`.
     - `provider` (Smartlead/SMTP).
     - `scope` (all leads / selected leads).
   - Status lifecycle: `created → queued → running → completed/failed`.

**Why this matters:**
- All future analytics and SIM rely on knowing exactly who received what, under which segment version.

---

### W0.v2 – Natural-Language Selection & ICP Hypothesis Capture

**Goal:** Remove the "SQL-UI" feeling for the user and connect audience selection to ICP hypothesis formation.

**Additions vs v1:**

1. **Chat-based filter builder**
   - Chat panel where the user types or dictates selection criteria.
   - AI coach parses the request into a structured set of company/employee filters.
   - The UI shows the generated filters, and the user can manually adjust them.

2. **ICP hypothesis drafts**
   - The original user query text is stored as an `icp_hypothesis` draft.
   - A structured hypothesis (market, roles, pains, value propositions) is generated by the coach and attached to `segment.version = 0`.

3. **Link with W1 (ICP profiles)**
   - The user can choose an existing `icp_profile` or create a new one via the sales_coach.
   - `segment` stores references to `icp_profile_id` and `icp_hypothesis_id`.

**UX principle:**
- The user speaks in natural language; the system turns that into filters + hypothesis, not the other way around.

---

### W0.v3 – Enrichment Entry Point (EXA / Parallel / Firecrawl)

**Goal:** Add enrichment after the audience is finalized but before generating/reviewing emails.

**Additions vs v2:**

1. **Enrich segment action**
   - At `segment.version = 1`, the user can trigger an "Enrich segment" action.
   - The system sends the list of companies/employees to integrations (EXA, Parallel, Firecrawl) via MCP/API.

2. **Data targets**
   - Enrichment results are written into:
     - `companies.company_research` / a dedicated research table.
     - `employees.ai_research_data` / a dedicated research table.

3. **Status & cost awareness**
   - Enrichment jobs have a status (pending/running/done/failed) and a rough cost estimate (number of requests, limits).

**Constraint:**
- Enrichment does not change segment membership; it only adds fields used for personalization and SIM.

---

### W0.v4 – SIM Entry Point (5-Contact Sample)

**Goal:** Allow testing the hypothesis and email tone on a small sample before mass sending.

**Additions vs v3:**

1. **Sample-based preview**
   - If `segment_members` < 5 → show all contacts.
   - If `segment_members` ≥ 5 → show aggregates + a random sample of 5 employees.
   - A "Show another 5" action can be triggered an unlimited number of times.

2. **SIM launch**
   - The user chooses "Run simulation" on the current sample of 5.
   - The SIM engine receives:
     - company and employee profile (including enrichment),
     - `icp_profile`, `icp_hypothesis`,
     - draft emails (intro + bump).

3. **SIM results display**
   - For each of the 5 employees, show:
     - a short persona/priority summary,
     - an interest/relevance score for the email,
     - likely behavior (read/ignore/reply/mark as spam).

**Relationship to Workflow 2:**
- W0.v4 does not implement SIM logic itself; it only creates the request and displays results returned by W2.

## 2. Workflow 1 – ICP & Email Coach

### W1.v0 – Monolithic Email Coach

**Current state:**
- A single large prompt does everything: clarifies ICP, understands the role, generates the email, chooses tone and CTA.
- There is no structured storage for `icp_profile` / `icp_hypothesis`.

---

### W1.v1 – Embedded ICP in Coach

**Goal:** Slightly improve the existing prompt without breaking UX.

**Characteristics:**
- The prompt emits a free-text description of ICP and hypothesis, but it is stored in a semi-structured way only.
- These data can be used for tags/analytics but not as full-fledged entities.

---

### W1.v2 – Full ICP Profiles & Hypotheses + Prompt Split

**Goal:** Make ICP and hypotheses first-class objects and prepare for agent-based execution.

**Components:**

1. **ICP profile formation (Step 0)**
   - A dedicated coaching flow forms an `icp_profile`:
     - industry, company type, size, region,
     - decision-makers and influencers,
     - typical pains and triggers.
   - The profile is stored and can be reused across segments.

2. **ICP hypothesis formation (Step 1)**
   - Based on the chosen `icp_profile`, a concrete hypothesis for the segment is formed:
     - what exactly is being tested (offer / angle / channel / role, etc.),
     - expected outcome (reply/call/demo),
     - funnel position (cold vs warm).
   - The `icp_hypothesis` is stored and linked to the segment.

3. **Individual email generation (Step 2)**
   - For each employee, taking into account:
     - `icp_profile`, `icp_hypothesis`,
     - role and research context,
     - chosen CTA,
     - pattern / Pattern Breaker mode,
     - the system generates a pair of emails (intro + bump).

4. **AI contracts (CORE)**
   - Introduce separate request/response types:
     - `GenerateIcpProfile`, `GenerateIcpHypothesis`, `GenerateEmailDrafts`.
   - Each response carries `pattern_id`, `coach_prompt_id`, and `email_type`.

## 3. Workflow 2 – SIM & Replies

### W2.v1 – Persona Simulation on 5-Contact Samples

**Goal:** Use enriched data and drafts to simulate recipient reactions.

**Inputs:**
- 5 employees (from W0.v4 sample).
- Linked company records, enrichment, LinkedIn/X.com data (where available).
- `icp_profile`, `icp_hypothesis`.
- Draft emails (intro + bump).

**Outputs:**
- For each employee:
  - synthesized persona summary,
  - relevance/fit score for the message,
  - likely behavior and probability of reply.
- For the segment/hypothesis as a whole:
  - aggregated "fit score" and an "expected reply rate band".

---

### W2.v2 – AI-Suggested Replies to Real Responses

**Goal:** Use the same contexts (ICP, patterns, SIM) to generate replies to real inbound emails.

**Inputs:**
- Original outbound draft,
- inbound reply from the lead,
- `icp_hypothesis`, `pattern_id`,
- deal/opportunity context (if present).

**Outputs:**
- 1–2 suggested replies,
- optional rationale/notes,
- explicit structure (key points, CTA).

**UX:**
- User sees a list of replies.
- Clicking a reply opens the suggested answer.
- User can accept, edit, or discard the suggestion.

## 4. Workflow AN – Analytics & Feedback Loop

### AN.v1 – Performance Analytics

**Goal:** Build analytics that explain "what works" across key dimensions.

**Dimensions:**
- ICP profiles,
- ICP hypotheses,
- segments,
- roles/positions,
- patterns / Pattern Breaker modes,
- `user_edited` (AI-only vs heavily edited drafts).

**Metrics:**
- emails sent,
- delivered,
- opened,
- replied,
- positive replies (interest/demo/call).

---

### AN.v2 – Prompt & Pattern Optimization

**Goal:** Close the loop: from results back into prompt and pattern changes.

**Capabilities:**
- Reports on underperforming and overperforming ICP hypotheses and patterns.
- Recommendations for adjusting agent prompts.
- A/B or at least time-based comparison of old/new prompt and pattern versions.
