# Отзыв Outreach: employee:repair-names

## Результат интеграционного теста

Команда протестирована на реальных данных (478 сотрудников в Supabase).

### Dry-run

```bash
pnpm --silent cli employee:repair-names --dry-run --error-format json
```

```json
{
  "scanned_count": 478,
  "candidate_count": 51,
  "fixable_count": 20,
  "skipped_count": 31,
  "updated_count": 0
}
```

### Анализ кандидатов

Команда разделила 51 кандидата на два уровня confidence:

**High confidence (20)** — все корректные. Проверено вручную каждого:
- "Инна Федина": first=Федина → Инна, last=Инна → Федина ✓
- "Андрей Моисеенков": first=Моисеенков → Андрей ✓
- "Елена Ачкасова": first=Ачкасова → Елена ✓
- ... все 20 — правильные fix-ы

**Low confidence (31)** — правильно пропущены. Это записи формата "Фамилия Имя" (Мамонов Константин), где first_name=Константин и last_name=Мамонов уже корректны. Команда верно определила, что менять их не нужно.

### Применение

```bash
pnpm --silent cli employee:repair-names --error-format json
```

Результат: `updated_count: 20`. Повторный `--dry-run` показал `fixable_count: 0`.

### Верификация

Контрольная проверка через Supabase REST API:
```
Инна Федина | first=Инна | last=Федина ✓
```

## Оценка

**Качество работы — отлично:**

1. **Точность классификации** — 100%. Ни одного ложного срабатывания среди high confidence, ни одного пропущенного среди low confidence.

2. **Безопасность** — low confidence (31 запись) не применяется. Это правильное решение: лучше пропустить сомнительный случай, чем испортить данные.

3. **Dry-run** — работает корректно, позволяет проверить план до применения.

4. **Идемпотентность** — повторный запуск показывает 0 fixable. Можно безопасно запускать регулярно.

## Предложения

1. **Добавить `--confidence high|low|all`** — чтобы оператор мог явно выбрать порог. Сейчас fixable = high only, что правильно по умолчанию, но явный фильтр был бы полезен.

2. **Интеграция в company:save-processed** — валидация first/last name при сохранении новых сотрудников, чтобы проблема не повторялась.

3. **Логирование** — при применении fix-ов записывать в metadata сотрудника что fix был применён (original_first, original_last, repaired_at, repair_source). Для audit trail.

## Статус

Gap #5 (first_name/last_name перепутаны) — **закрыт**. 20 записей исправлены, 0 оставшихся.
